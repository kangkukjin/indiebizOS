# 작업계획서 가이드 (create_plan / execute_plan)

여러 에이전트가 **순서대로** 협력해야 하는 복잡한 작업을 위한 시스템.
한 에이전트가 혼자 할 수 없거나, 여러 프로젝트의 전문성이 각각 필요한 경우에 사용한다.

---

## 언제 쓰는가

| 상황 | 적합한 도구 |
|------|------------|
| 혼자서 깊이 파야 하는 작업 | Goal 시스템 |
| 한 턴 안에서 자기 작업 정리 | todo |
| **여러 에이전트의 전문성이 순서대로 필요** | **작업계획서** |
| 시간 기반 실행 | schedule |

예: "이란 전쟁 보고서"를 쓰려면 뉴스 수집(컨텐츠) → 경제 분석(투자) → 종합 정리(컨텐츠) 순서로 각 전문 에이전트가 필요하다. 이때 작업계획서를 쓴다.

---

## 핵심 원칙

1. **순차 실행**: 1단계 완료 → 2단계 시작 → 3단계 시작. 병렬 아님.
2. **시스템이 관리**: 계획서를 파싱하고, 다음 단계를 찾고, 담당 에이전트에게 전달하는 건 시스템이 한다. 에이전트가 직접 계획서를 읽고 판단하는 게 아님.
3. **에이전트는 자기 단계만 받는다**: 계획서 전체를 넘기지 않고, 해당 단계의 지시사항만 전달.
4. **의도를 전달하고 실행은 맡긴다**: description에 자연어로 "뭘 해야 하는지"를 적고, "어떻게 하는지"(IBL 코드 등)는 담당 에이전트가 판단한다.

---

## 작성 방법

### create_plan 호출

```
[self:create_plan]{
  file: "outputs/report_plan.md",
  title: "이란 전쟁 향후 전개 보고서",
  goal: "이란 전쟁의 향후 전개를 분석하는 보고서를 작성",
  steps: [
    {
      agent_project_id: "컨텐츠",
      agent_id: "agent_001",
      title: "최신 뉴스 수집",
      description: "이란 전쟁 관련 최신 뉴스와 핵심 쟁점을 수집하고 요약해주세요."
    },
    {
      agent_project_id: "투자",
      agent_id: "agent_001",
      title: "경제적 파급효과 분석",
      description: "이전 단계의 뉴스 요약을 바탕으로 경제적 파급효과와 투자 시사점을 분석해주세요."
    },
    {
      agent_project_id: "컨텐츠",
      agent_id: "agent_001",
      title: "종합 보고서 작성",
      description: "이전 단계들의 결과를 종합하여 보고서를 작성해주세요."
    }
  ]
}
```

### steps 각 항목의 필드

| 필드 | 필수 | 설명 |
|------|------|------|
| agent_project_id | ✅ | 담당 에이전트의 프로젝트 ID |
| agent_id | ✅ | 담당 에이전트 ID |
| title | ✅ | 단계 제목 |
| description | ✅ | **자연어**로 해야 할 일 설명 |
| pipeline | - | IBL 코드 (선택. 보통 비워두고 에이전트에게 맡김) |
| on_failure | - | 실패 시 지시 (기본: "보고하고 중단") |
| max_retries | - | 최대 재시도 횟수 (기본: 2) |

## description 작성 규칙

### ✅ 올바른 예 — 자연어로 의도 전달

```
description: "이란 전쟁 관련 최신 뉴스를 검색하고 핵심 쟁점을 정리해주세요."
description: "이전 단계의 분석 결과를 바탕으로 투자 포트폴리오 조정 방안을 제안해주세요."
description: "수집된 데이터를 종합하여 A4 2페이지 분량의 보고서를 작성해주세요."
```

### ❌ 잘못된 예 — IBL 코드를 미리 작성

```
description: "[sense:search_news]{query: '이란 전쟁', count: 10} >> [self:file_write]{...}"
description: "[sense:kr_stock_price]{symbol: '005930'} >> [data:analyze]{...}"
```

왜 잘못인가? 담당 에이전트는 자기 도구, 자기 노트, 자기 맥락이 있다. 위임하는 측이 미리 IBL 코드를 짜는 건 상대의 전문성을 무시하는 것이다.

---

## 실행 방법

계획서 생성 후 바로 실행:

```
[self:create_plan]{file: "outputs/plan.md", ...}
[self:execute_plan]{file: "outputs/plan.md"}
```

또는 스케줄에 연결:

```
[self:schedule]{at: "09:00", pipeline: "[self:execute_plan]{file: 'outputs/plan.md'}"}
```

---

## 실행 흐름 (시스템 내부)

```
execute_plan 호출
    ↓
계획서 파싱 → 상태가 "대기"인 첫 번째 단계 찾기
    ↓
해당 단계의 지시사항만 추출 → 담당 에이전트 창 열기 → 메시지 전달
    ↓
에이전트가 작업 수행 (채팅창에 실시간 표시)
    ↓
에이전트 완료 → 시스템이 상태를 "완료"로 업데이트
    ↓
다음 "대기" 단계가 있으면 → 자동으로 다음 에이전트에게 전달
    ↓
모든 단계 완료 시 종료
```

에이전트가 받는 메시지 예시:

```
## 작업 지시: 최신 뉴스 수집
(작업계획서 'report_plan' — 단계 1/3)

### 해야 할 일
이란 전쟁 관련 최신 뉴스와 핵심 쟁점을 수집하고 요약해주세요.
```

이전 단계의 결과가 있으면 "이전 단계 결과" 섹션도 함께 전달된다.

---

## 상태 마커

| 마커 | 의미 |
|------|------|
| `[ ] 대기` | 아직 실행 안 됨 |
| `[~] 진행중` | 에이전트가 작업 중 |
| `[v] 완료` | 성공적으로 완료 |
| `[x] 실패` | 실행 실패 |

---

## 주의사항

- **execute_plan 없이 직접 위임하지 말 것**: 계획서를 만들어놓고 `delegate_project`로 병렬 위임하면 순서가 깨진다. 반드시 `execute_plan`으로 실행해야 순차 실행과 상태 추적이 된다.
- **한 번에 하나씩**: 시스템이 한 단계가 끝나야 다음 단계를 시작한다. 병렬 실행이 필요하면 작업계획서가 아닌 `&` 연산자나 직접 위임을 사용한다.
- **실패 시**: 해당 단계가 "실패"로 표시되고 다음 단계로 진행하지 않는다.
