# AI 동향 보고서 가이드 (AI Trend Report)

사용자(강국진)에게 **매일 1회** AI 분야의 주목할 동향을 요약 보고하는 정기 보고서를 작성하는 절차다. 핵심은 **"매번 새로 쓰는 것이 아니라, 직전 보고서를 찾아 읽고 → 새로 조사한 자료를 더해 → 무엇이 새로워졌는가를 짚으며 업데이트"** 하는 것이다.

이 보고서는 세 섹션으로 구성된다:
1. **투자·경제 섹션** — AI 분야의 투자·자금·시장·기업 실적·정책 동향
2. **기술·연구 동향 섹션** — 새 모델·논문·연구 성과·오픈소스·제품 출시
3. **AI 활용 사례 섹션** — 사람들이 AI를 실제로 써서 해낸 일들 중, 새롭고 파급력 있는 사례를 골라 소개

---

## 0. 저장 위치 규약 (반드시 고정)

과거 보고서를 찾으려면 저장 위치가 한 곳으로 고정되어 있어야 한다. **예외 없이** 아래를 따른다.

- **고정 폴더**: `/Users/kangkukjin/Desktop/AI/indiebizOS/outputs/ai_trend_reports/`
- **파일명 규약**: `ai_trend_report_YYYY-MM-DD.md` (예: `ai_trend_report_2026-06-17.md`)
  - 날짜는 작성 당일 기준. 현재 날짜는 시스템 프롬프트 "현재 시점" 또는 `[self:time]`으로 확인.
  - 날짜 기반 파일명이므로 파일명을 사전순 정렬하면 곧 시간순 정렬이 된다(최신 보고서 찾기의 핵심).
- 폴더가 없으면 첫 작성 시 `[self:write]`가 자동 생성한다(별도 mkdir 불필요).

## 1. 직전 보고서 찾아 읽기

새 보고서를 쓰기 전, **항상 가장 최근 보고서를 먼저 찾아 읽는다.**

### 1-1. 최신 보고서 파일 찾기

```
[self:file_find]{path: "/Users/kangkukjin/Desktop/AI/indiebizOS/outputs/ai_trend_reports/", pattern: "ai_trend_report_*.md"}
```

- 결과 목록에서 **파일명 사전순으로 가장 마지막** 파일이 최신 보고서다 (날짜 파일명 규약 덕분).
- 결과가 비어 있으면 → **이전 보고서가 없는 첫 작성.** 이 경우 1-2, 2단계(델타 비교)를 건너뛰고 처음부터 새로 작성한다(섹션 구조와 조사 방법은 동일하게 따른다).

### 1-2. 직전 보고서 본문 읽기

```
[self:read]{path: "<위에서 찾은 최신 파일의 절대경로>"}
```

- 직전 보고서에서 **이미 다룬 주제·수치·고유명사(기업/모델/논문명)**를 파악한다. 이것이 "무엇이 새로운가"를 판단하는 기준선이 된다.
- 직전 보고서의 "지켜볼 점/전망" 항목이 있으면, 그것이 이번에 실제로 어떻게 됐는지 확인 대상으로 삼는다.

## 2. 새 자료 조사 (섹션별)

조사는 **투자·기술 두 섹션 각각 최소 3회 이상, 전체적으로 영어 검색 2회 이상**을 포함한다(활용 사례 섹션은 추가로 살핀다). 1~2회 검색 후 보고서로 직행하지 않는다.

> ⚠️ **파이프라인 금지**: 검색과 보고서 작성을 `>>`로 엮지 않는다. IBL 검색 액션을 **하나씩** 호출하고, 결과를 읽고 판단한 다음 추가 검색 여부를 결정한다. 검색 → 분석 → 추가 검색 → 종합의 매 단계에서 직접 생각한다. (심층연구 가이드 `deep_research.md`의 원칙을 따른다.)

### 2-1. 투자·경제 섹션 조사

사용할 IBL 검색 액션:
- `[sense:search_news]{query: "AI 투자"}` — 한국어 뉴스 (Google News)
- `[sense:search_news]{query: "AI investment funding"}` — 영어 뉴스
- `[sense:search_naver]{query: "AI 반도체 투자", type: "news"}` — 한국 출처 강화
- `[sense:search_ddg]{query: "AI chip market Nvidia earnings"}` — 글로벌 웹
- 필요 시 종목 수치 확인: `[sense:stock]{op: "quote", ticker: "NVDA"}`, `[sense:stock]{op: "news", ...}`
- 핵심 기사는 스니펫에 그치지 말고 `[sense:crawl]{url: "..."}`로 원문을 확인한다.

조사 대상 예: AI 기업 자금조달·M&A, 반도체(엔비디아/HBM 등) 수급과 실적, 빅테크 AI 투자 발표, AI 관련 정책·규제, 주가·시장 반응.

### 2-2. 기술·연구 동향 섹션 조사

사용할 IBL 검색 액션:
- `[sense:search_news]{query: "AI 신모델 출시"}` / `[sense:search_news]{query: "AI model release"}`
- `[sense:search_ddg]{query: "new LLM benchmark state of the art"}` — 영어 웹
- `[sense:paper]{op: "search", query: "large language model", source: "arxiv"}` — 최신 논문
- 중요한 논문/발표는 `[sense:crawl]{url: "..."}`로 원문 확인.

조사 대상 예: 새 모델·버전 출시, 벤치마크 갱신, 주목 논문, 오픈소스 릴리스, 새 기법·아키텍처, 멀티모달/에이전트/로보틱스 진전.

### 2-3. AI 활용 사례 섹션 조사

사람들이 AI를 **실제로 써서** 새로운 일을 해낸 사례를 찾는다. 모델·자금 뉴스가 아니라, AI가 현실에서 무엇을 가능하게 했는가에 초점을 둔다.

사용할 IBL 검색 액션:
- `[sense:search_news]{query: "AI 활용 사례"}` / `[sense:search_news]{query: "how people use AI 2026"}`
- `[sense:search_ddg]{query: "impressive AI use case real world 2026"}` — 영어 웹
- `[sense:search_naver]{query: "AI로 만든", type: "blog"}` — 개인·현장의 실사용담
- 인상적인 사례는 스니펫에 그치지 말고 `[sense:crawl]{url: "..."}`로 원문을 확인한다.

조사 대상 예: 과학·의료(신약·진단·발견), 창작(영상·음악·코드·디자인), 산업·업무 자동화, 교육, 행정·공공 서비스, 개인의 기발한 활용 — 분야는 가리지 않는다.

**선별 기준 — 아무 사례나 넣지 않는다.** 다음 둘을 모두(또는 어느 한쪽을 강하게) 만족하는 것만 고른다:
- **신규성**: 지금까지 보지 못한 새로운 방식·영역의 활용인가. 이미 흔한 챗봇 Q&A·요약 같은 건 제외한다.
- **파급력**: 많은 사람·산업에 영향을 주거나, 일하는 방식·판도를 바꿀 잠재력이 있는가.

신규성·파급력이 큰 사례가 **없는 날이면 이 섹션은 비워도 된다**(억지로 채우지 않는다 — 분량 채우기는 안티패턴이다).

### 2-4. 소스 신뢰도 원칙 (필수)

뉴스·웹 검색은 **언론 보도 수준이며 1차 자료가 아니다.** 증권사 리서치·논문 원문에 직접 접근하는 데는 한계가 있다. 따라서:

- **복수 소스 교차 검증**: 핵심 수치·주장은 서로 독립된 2개 이상 출처에서 확인되기 전에는 단정하지 않는다. 한 곳에만 나오면 "~에 따르면"으로 출처를 명시해 단정과 구분한다. 두 소스가 어긋나면 그 불일치를 그대로 적는다.
- **출처 표기**: 모든 핵심 항목에 출처(매체명/URL)를 붙인다. 이 세션에서 **실제로 호출한** 검색·크롤 결과에 등장한 자료만 인용한다. 호출하지 않은 권위 기관(IMF·World Bank 등)을 "있어 보여서" 인용하지 않는다(출처 날조 금지).
- **사실/추정 구분**: "~로 보도되었다"(사실)와 "~일 가능성이 있다"(추정)를 명확히 나눈다.

### 2-5. 자료 신선도(시점) 검증 (필수)

이 보고서의 생명은 **최신성**이다. 그런데 `search_news`·`search_ddg`는 발행일과 무관하게 **몇 달~몇 년 전 기사도 섞어** 반환한다. 시점을 검증하지 않으면 낡은 사건이 "오늘의 동향(NEW)"으로 둔갑한다(실제로 첫 보고서에서 6개월 전 'Dec 2025 AI주 급락'이 NEW 항목으로 들어가는 사고가 있었다).

- **각 자료의 발행 시점을 반드시 확인한다.** 검색 결과 제목/스니펫에 날짜가 없으면 `[sense:crawl]`로 원문 발행일을 확인하고, 끝내 확인 불가하면 "시점 미상"으로 표시하거나 제외한다.
- **"NEW"·"현재 동향"으로 올릴 자료는 보고 시점(오늘) 기준 대략 최근 1~2주 이내 사건으로 제한**한다. 그보다 오래된 자료는 (a) 최신 흐름을 설명하는 배경 맥락으로만 한두 줄 쓰거나 (b) 제외한다 — 결코 NEW 라벨을 붙이지 않는다.
- **자가 점검**: 'NEW'를 붙인 모든 항목에 대해 "이게 정말 며칠~몇 주 내 일인가?"를 자문한다. 크롤한 문서의 제목·URL에 'Dec 2025', '2025' 같은 시점 표기가 있으면 그 시점을 기준으로 판단한다.
- 검색 단계에서 쿼리에 현재 연·월(예: "AI 투자 2026")을 넣으면 최신 자료 비중을 높일 수 있다.

## 3. 업데이트(델타) 서술 절차 — "무엇이 새로운가"

이 보고서의 정체성은 **델타(변화분)**다. 직전 보고서(1단계)와 새 조사(2단계)를 대조하여 다음을 식별·서술한다:

1. **새로 등장한 것 (NEW)**: 직전 보고서에 없던 사건·발표·수치.
2. **달라진 것 (CHANGED)**: 직전에 다뤘으나 상황이 진전/반전된 것 (예: "지난번 루머였던 X가 공식 발표됨", "주가가 N% 추가 상승").
3. **이어지는 것 (ONGOING)**: 계속 주목할 흐름이되 변화가 작은 것은 짧게.
4. **예측 점검**: 직전 보고서의 전망이 맞았는지/빗나갔는지 한 줄로 짚는다.

각 항목 앞에 직전 대비 무엇이 바뀌었는지를 명시한다(예: "(NEW) ...", "(지난 보고서 대비 진전) ..."). 변화가 없는 뻔한 배경 설명으로 분량을 채우지 않는다.

## 4. 보고서 출력 형식

아래 골격으로 `[self:write]`를 사용해 **0단계 규약 경로**에 저장한다. 보고서는 AI가 직접 작성한다(`engines:newspaper` 자동 생성 도구에 위임하지 않는다 — 신문이 아니라 분석 보고서다).

```markdown
# AI 동향 보고서 — YYYY-MM-DD

> 직전 보고서: ai_trend_report_<직전날짜>.md (없으면 "첫 보고서")

## 한눈에 보는 변화 (TL;DR)
- (가장 중요한 변화 3~5개를 불릿으로. 투자·기술·활용 사례에서)

## 1. 투자·경제
### 새로운 것 (NEW)
- ...
### 달라진 것 (CHANGED)
- ...
### 이어지는 흐름 (ONGOING)
- ...

## 2. 기술·연구 동향
### 새로운 것 (NEW)
- ...
### 달라진 것 (CHANGED)
- ...
### 이어지는 흐름 (ONGOING)
- ...

## 3. 주목할 AI 활용 사례
- (새롭고 파급력 있는 실사용 사례 1~3개. 누가·무엇을·어떻게 했고, 왜 새롭거나 의미 있는지 한두 줄로. 해당 사례가 없으면 이 섹션 자체를 생략)

## 지켜볼 점 (다음 보고서 점검 대상)
- ...

## 출처
- [매체명](URL) — [원문] 또는 [검색결과] 등급 표기
```

저장 후 사용자에게 저장 경로(절대경로)와 핵심 변화 요약을 보고한다. 사용자가 원하면 `[self:output]{op: "gui"}`로 화면에 표시하거나 HTML로 함께 렌더할 수 있다.

## 5. 매일 1회 자동 작성 (스케줄 등록)

이 보고서는 매일 1회 정기 작성하도록 트리거로 등록할 수 있다. 등록 방법은 **트리거·스위치 가이드(`trigger.md`)** 또는 **캘린더 & 스케줄러 가이드(`calendar.md`)**를 참조한다.

- 권장: schedule 타입 트리거, 매일 아침 1회(예: 08:00).
- 트리거가 실행할 작업은 단순히 **"AI 동향 보고서 써줘"** 의도를 실행 에이전트에 전달하는 것 — 그러면 이 가이드가 검색·로드되어 위 0~4 절차대로 직전 보고서를 참고한 업데이트 보고서가 생성된다.
- 등록 예시 의도: `[self:trigger]{op: "create", name: "AI 동향 보고서 일일", schedule: ..., pipeline: ...}` — 구체적 파라미터는 `trigger.md`를 읽고 구성한다.
- 작성 완료 시 `[self:notify_user]`로 사용자에게 알림을 보내면 좋다.

> 스케줄 등록은 사용자가 명시적으로 요청할 때 수행한다. 가이드 단계에서는 등록까지 하지 않는다.

## 체크리스트 (보고서 작성 시)

- [ ] 0단계 고정 경로에서 최신 보고서를 찾았는가 (`file_find`)
- [ ] 직전 보고서를 읽고 기준선을 파악했는가 (없으면 첫 작성으로 분기)
- [ ] 투자·기술 두 섹션을 각각 조사했고(활용 사례는 해당 시), 영어 검색 2회 이상 포함했는가
- [ ] 새롭고 파급력 있는 AI 활용 사례를 살폈는가 (신규성·파급력 큰 게 없으면 생략 가능, 2-3절)
- [ ] 핵심 수치를 복수 소스로 교차 검증하고 출처를 표기했는가
- [ ] 인용한 각 자료의 발행 시점을 확인하고, 오래된 자료를 'NEW/현재 동향'으로 오인하지 않았는가 (2-5절)
- [ ] "무엇이 새로운가"(NEW/CHANGED/ONGOING)를 명시했는가
- [ ] 0단계 규약 경로에 날짜 파일명으로 저장했는가
- [ ] 사용자에게 절대경로와 핵심 변화를 보고했는가
