# 출판 프로젝트 관리 가이드

책을 쓰거나, 원고를 관리하거나, 출판 관련 요청을 받으면 이 가이드를 따른다.

---

## 핵심 개념

- **출판 프로젝트(book)**: 한 권의 책에 해당하는 작업 단위
- **플랜(plan)**: 책의 구조 설계서 — 하나의 책에 여러 플랜을 시도할 수 있다
- **버전(version)**: 같은 플랜 내에서 다르게 집필한 원고 — 피드백 반영, 문체 변경, 재작성 등
- **조각글(fragment)**: 아직 챕터에 배치되지 않은 원고 조각 (플랜/버전과 무관하게 공유)
- **챕터 원고(draft.md)**: 각 챕터의 실제 원고
- **참고자료(references/)**: 논문, 스크랩, URL 메모 등 (공유)
- **진척 기록(progress.md)**: 프로젝트 상태, 결정 사항, 변경 이력

---

## 폴더 구조

각 출판 프로젝트는 `출판사` 프로젝트의 outputs 폴더 아래에 위치한다.

```
projects/출판사/outputs/books/
└── {book_id}/
    ├── book_registry.yaml      # 메타데이터 (제목, 저자, 상태, 현재 활성 플랜/버전)
    ├── progress.md             # 진척 기록 (상태, 결정, 이력)
    ├── fragments/              # 미배치 조각글 (모든 플랜/버전에서 공유)
    │   ├── 20260407_카페에서.md
    │   └── ...
    ├── references/             # 참고자료 (공유)
    │   ├── papers/
    │   ├── clippings/
    │   └── urls.md
    ├── images/                 # 공용 이미지 (공유)
    └── plans/                  # 플랜별 폴더
        ├── plan_v1/
        │   ├── plan.md         # 이 플랜의 구조 설계서 (목차 + 챕터별 요약)
        │   ├── v1/             # 첫 번째 집필 버전
        │   │   └── chapters/
        │   │       ├── ch01_제목/
        │   │       │   ├── draft.md
        │   │       │   ├── notes.md
        │   │       │   └── images/
        │   │       └── ch02_제목/
        │   └── v2/             # 같은 플랜, 다른 시도 (피드백 반영 등)
        │       └── chapters/
        └── plan_v2/            # 구조를 바꾼 새 플랜
            ├── plan.md
            └── v1/
                └── chapters/
```

### 플랜 vs 버전
- **새 플랜**: 목차/구조 자체를 바꿀 때 (챕터 순서 변경, 파트 재구성, 주제 범위 변경)
- **새 버전**: 같은 구조에서 내용을 크게 다시 쓸 때 (문체 변경, 피드백 반영, 관점 전환)
- 작은 수정은 새 버전이 아니라 기존 draft.md를 직접 수정하면 된다

## 파일 형식

### plan.md (플랜별 구조 설계서)

각 플랜 폴더 안에 위치. 이 파일이 해당 플랜의 설계도.

```markdown
# 책 제목

## 개요
한두 문단으로 이 책이 무엇인지, 누구를 위한 것인지.

## 구성 방침
- 각 챕터 예상 분량: 약 3000~5000자
- 문체: 대화체 / 학술체 / 에세이체
- 문체 참고 원고: (파일 경로 또는 URL)
- 특수 요소: 각 챕터 끝에 "핵심 정리" 박스 포함
- 부록: 용어 해설, 참고문헌 목록

## 목차

### Part 1: 파트 제목
#### Chapter 1: 챕터 제목
- **의도**: 이 챕터가 왜 필요한지
- **핵심 내용**: 다룰 내용 요약 (3~5줄)
- **참고**: 관련 조각글이나 참고자료 언급
- **상태**: 미착수 / 초안작성중 / 초안완료 / 검토중 / 완료

#### Chapter 2: 챕터 제목
- **의도**: ...
- **핵심 내용**: ...
- **특수 요소**: 중간에 "역사 박스" 삽입
- **상태**: 미착수

### Part 2: 파트 제목
...

### 부록
#### 부록 A: 용어 해설
#### 부록 B: 참고문헌
```

### book_registry.yaml (메타데이터)

```yaml
id: my-first-book
title: "책 제목"
subtitle: "부제목"
author: "저자명"
status: drafting  # planning / drafting / reviewing / editing / completed
created_at: "2026-04-07"
updated_at: "2026-04-07"
target_length: "200페이지"
genre: "비소설"
description: "이 책에 대한 한 줄 설명"
active_plan: plan_v1    # 현재 작업 중인 플랜
active_version: v1      # 현재 작업 중인 버전
```

### progress.md (진척 기록)

에이전트가 매번 질문하지 않도록, 결정 사항과 현재 상태를 기록한다.

```markdown
# 진척 기록

## 현재 상태
- 활성 플랜: plan_v1
- 활성 버전: v2 (v1에서 피드백 반영하여 재작성)
- 전체 진행률: 3/12 챕터 초안 완료
- 현재 작업: Chapter 4 초안 작성 중
- 다음 할 일: Chapter 4 완료 후 Chapter 2 검토

## 플랜/버전 이력
- [2026-04-07] plan_v1 생성 (12개 챕터)
- [2026-04-10] plan_v1/v1 → 초안 완료, 사용자 피드백 수렴
- [2026-04-11] plan_v1/v2 생성 (v1 기반, 문체를 대화체로 전환)
- [2026-04-15] plan_v2 생성 (Part 1, 2 순서 교체, 새 챕터 3개 추가)

## 결정 사항
- [2026-04-07] 문체는 대화체로 통일
- [2026-04-07] 각 챕터 끝에 요약 박스 넣기로 결정
- [2026-04-11] v1의 문체가 딱딱하다는 피드백 → v2에서 대화체로 전환

## 변경 이력
- [2026-04-07] 프로젝트 생성, 초기 목차 12개 챕터 구성
- [2026-04-08] Chapter 1 초안 완료
- [2026-04-08] 조각글 "카페에서" → Chapter 3에 배치
```

## 워크플로우

### A. 새 책 시작하기

```
1. book_registry 생성 (제목, 저자, 장르 등 메타데이터)
2. plans/plan_v1/ 폴더 생성
3. 사용자와 대화하며 plan.md 작성 (목차 + 챕터별 의도/내용)
4. plans/plan_v1/v1/chapters/ 폴더 구조 생성
5. progress.md 초기화
6. book_registry.yaml에 active_plan: plan_v1, active_version: v1 기록
```

**핵심**: plan.md를 먼저 충분히 잡아야 한다. 사용자와 목차를 반복해서 다듬는 것이 가장 중요한 단계.

### B. 원고 작성 (챕터 쓰기)

```
1. book_registry.yaml에서 활성 플랜/버전 확인
2. 해당 plan.md에서 챕터의 의도/핵심 내용 확인
3. fragments/에서 관련 조각글 탐색
4. references/에서 관련 참고자료 확인
5. 활성 버전의 chapters/chXX/draft.md 작성 또는 보완
6. plan.md에서 해당 챕터 상태 업데이트
7. progress.md에 작업 내용 기록
```

### C. 조각글 수집

사용자가 어떤 형식으로든 텍스트를 주면:
```
1. 형식 판별 (텍스트 / .pages / .docx / .pdf / 이미지 / URL)
2. 텍스트 추출
   - .pages/.docx: textutil -convert txt {파일경로}
   - .pdf: 텍스트 추출
   - 이미지: OCR
   - URL: 웹 크롤링으로 내용 발췌
3. fragments/에 날짜_제목.md로 저장
4. 관련 챕터가 명확하면 plan.md에 참고 메모 추가
```

**출전 기록 필수**: 모든 조각글에는 출처를 반드시 기록한다.
```markdown
---
title: 조각글 제목
date: 2026-04-07
source: 사용자 직접 작성 / 블로그 URL / 논문명 / 파일명 등
source_type: original / blog / paper / book / web / conversation
---

(본문)
```
- `original`: 사용자가 직접 쓴 글
- `blog`, `paper`, `book`, `web`: 외부 출처에서 발췌/참고한 경우 — 원본 URL이나 서지정보를 source에 명시
- `conversation`: 에이전트와의 대화에서 나온 내용

챕터 draft.md에 조각글 내용을 반영할 때도, 외부 출처 기반이면 각주나 참고표시를 남긴다. 이는 나중에 부록 참고문헌 목록 작성에도 활용된다.

### D. 조각글 → 챕터 배치

```
1. fragment 내용 읽기
2. 활성 플랜의 plan.md와 대조하여 적합한 챕터 결정
3. 활성 버전의 해당 챕터 draft.md에 내용 통합 (단순 붙여넣기가 아닌 문맥에 맞게 편집)
4. fragment 파일을 삭제하거나 "[배치완료: plan_v1/v2/ch03]" 표시
5. progress.md 업데이트
```

### E. 새 버전 만들기

같은 플랜에서 크게 다시 쓰고 싶을 때:
```
1. 현재 버전(예: v1)의 chapters/ 전체를 새 버전(v2)으로 복사
2. 또는 빈 v2/chapters/를 만들어 처음부터 다시 쓰기
3. book_registry.yaml의 active_version을 v2로 변경
4. progress.md의 "플랜/버전 이력"에 왜 새 버전을 만들었는지 기록
```

**이전 버전은 그대로 보존된다.** 나중에 비교하거나 되돌릴 수 있다.

### F. 새 플랜 만들기

목차/구조 자체를 바꾸고 싶을 때:
```
1. plans/plan_v2/ 폴더 생성
2. 새 plan.md 작성 (기존 plan.md를 복사해서 수정하거나 처음부터)
3. plan_v2/v1/chapters/ 폴더 구조 생성
4. book_registry.yaml의 active_plan을 plan_v2, active_version을 v1로 변경
5. progress.md에 플랜 변경 이유 기록
```

**이전 플랜도 그대로 보존된다.**

### G. 구조 변경 (현재 플랜 내)

사용자와 대화하며 목차를 수정할 때 (작은 변경):
```
1. 현재 활성 플랜의 plan.md 수정 (챕터 추가/삭제/순서변경/내용변경)
2. 필요시 활성 버전의 chapters/ 폴더 이름 변경/추가/삭제
3. progress.md의 "변경 이력"에 무엇을 왜 바꿨는지 기록
```

### H. 참고자료 활용

```
1. 사용자가 논문/블로그/텍스트를 제공
2. references/에 저장 (papers/, clippings/, 또는 urls.md에 추가)
3. 핵심 내용을 발췌하여 해당 챕터의 notes.md에 정리
4. draft.md 작성 시 notes.md를 참고하여 반영
```

### I. 원고 합본 (Export)

특정 플랜/버전의 원고를 합본한다:
```
1. 대상 플랜/버전 확인 (기본: 활성 플랜/버전)
2. plan.md의 목차 순서대로 해당 버전의 chapters/chXX/draft.md를 합침
3. 마크다운 하나의 파일로 출력
4. 필요시 pandoc으로 docx/PDF 변환:
   pandoc output.md -o output.docx
   pandoc output.md -o output.pdf --pdf-engine=xelatex -V mainfont="AppleGothic"
```

## 문체 참고 원고 (Style Reference)

AI가 쓴 글이 아닌, 사용자의 문체로 쓰인 글이 되려면 문체 참고가 필수다.

### 설정 방법
plan.md의 "구성 방침"에 문체 참고 원고를 지정한다:
```markdown
- 문체 참고 원고: fragments/내_에세이_샘플.md
```
또는 블로그 URL을 지정할 수도 있다:
```markdown
- 문체 참고 원고: https://blog.example.com/my-post
```

### 참고 원고가 있을 때의 작성 규칙
1. **초안 작성 전**: 반드시 문체 참고 원고를 먼저 읽는다
2. **문체 요소 파악**: 문장 길이, 어미 스타일(~다/~요/~죠), 비유 방식, 단락 구성, 전문용어 사용 수준 등을 분석
3. **작성 시 적용**: 내용은 plan.md에 따르되, 문체는 참고 원고를 따라 쓴다
4. **챕터별 다른 문체도 가능**: 특정 챕터에 별도의 문체 참고를 지정할 수 있다
   ```markdown
   #### Chapter 5: 인터뷰
   - **문체 참고**: fragments/인터뷰체_샘플.md
   ```

### 참고 원고가 없을 때
- 사용자가 직접 쓴 조각글(source_type: original)이 있으면, 그 중 가장 긴 것을 문체 참고로 활용
- 그것도 없으면 사용자에게 "문체 참고할 글을 하나 주시면 그 톤으로 쓰겠습니다"라고 요청

## 저자의 사유 방식 및 서술 원칙 (AI 작법 강제 지침)

AI가 초안을 작성할 때, 위키백과식의 평이한 정보 나열을 엄격히 금지하며 다음의 원칙을 반드시 준수한다.

### 1. 단호한 1인칭 주관과 통찰 (Opinionated Writing)
- "일반적으로 ~라고 알려져 있다", "~장단점이 있다" 식의 양비론적, 중립적 서술을 금지한다.
- "나는 ~라고 생각한다", "~은 결국 ~하게 될 것이다"와 같이 저자의 확고한 철학과 미래에 대한 통찰을 단호한 어조로 단언하라.
- 기술의 겉모습(기능)이 아니라, 그 기술이 인간과 노동의 본질을 어떻게 바꾸는지(의미)에 집중하라.

### 2. 팩트와 철학의 분리 (팩트박스 의무화)
- 본문은 오직 저자의 주관적 해석, 비유, 흐름을 이어가는 데만 집중한다.
- 연도, 역사적 배경, 인물, 기술적 정의, 수치 등 객관적이고 건조한 정보는 본문 흐름을 끊지 않도록 반드시 `--- **팩트박스: [주제]** ---` 형식으로 분리하여 삽입하라.

### 3. 생활 밀착형 직관적 비유 (Metaphorical Explanation)
- 어려운 기술적 개념을 설명할 때는 반드시 독자가 즉각적으로 이해할 수 있는 물리적 세계의 비유를 하나 이상 포함하라.
- 문학적 장식("잠들어 있는 지능의 엔진")이 아닌, 구체적이고 직관적인 비유("바보라도 핵무기 버튼을 누르면 엄청난 일이 일어난다", "투명인간이 PC 앞에 앉아서 일하는 것처럼")를 사용하라.

### 4. 대화체 기반의 구어적 리듬감
- '-다/한다' 체를 기본으로 하되, 독자에게 직접 강연하거나 대화하듯 호흡을 조절하라. ("~라고 할 수 있다", "~라는 것이다", "예를 들어 ~를 생각해 보자")

### 5. 축약 금지 — AI의 역할은 확장이다
- **AI의 역할은 저자의 원고를 축약하는 것이 아니라 확장하는 것이다.** 저자가 쓴 원고는 뼈대이고, AI는 그 뼈대에 살을 붙인다.
- 저자의 주장을 검색과 조사로 뒷받침하고, 저자가 간략히 넘어간 부분을 더 깊이 파고들며, 팩트박스에 넣을 자료를 조사해서 채운다.
- **결과물은 반드시 원고보다 길고 깊어야 한다.** 원고를 "깔끔하게 정리"하는 것은 요약 AI이지 작가 AI가 아니다.
- 원고의 각 소주제(섹션)는 최소 500자 이상으로 충분히 전개할 것.
- "~가 있다"로 끝내지 말고, "왜 중요한지, 한계는 무엇인지, 그래서 어떻게 되는지"까지 전개할 것.
- 저자가 예시를 들었으면 추가 예시를 검색해서 보강하라. 저자가 역사를 언급했으면 구체적 날짜와 인물을 조사해서 팩트박스로 만들라.

## 핵심 원칙

1. **plan.md가 진실의 원천**: 목차, 상태, 방향 모두 여기에. 작업 전 반드시 활성 플랜의 plan.md를 읽는다.
2. **progress.md로 기억 유지**: 에이전트가 바뀌거나 대화가 끊겨도 맥락을 잃지 않는다. 활성 플랜/버전도 여기서 확인.
3. **사용자 입력 형식 무관**: 뭘 던져도 받는다. 변환은 에이전트의 몫.
4. **적극적 작성**: 비평만 하지 않는다. 직접 초안을 쓰고, 사용자의 피드백을 반영한다.
5. **사용자의 문체로 쓴다**: 문체 참고 원고를 반드시 확인하고, 사용자의 글쓰기 스타일을 따라 쓴다.
6. **구조 먼저, 글 나중에**: plan.md를 충분히 다듬은 후 원고 작성에 들어간다.
7. **이미지도 관리**: 챕터별 images/ 또는 공용 images/에 정리. plan.md에 이미지 계획도 기록 가능.
8. **버전은 보존한다**: 새 버전/플랜을 만들 때 이전 것을 삭제하지 않는다. 사용자가 명시적으로 삭제를 요청할 때만 삭제.
