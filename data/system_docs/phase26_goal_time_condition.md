# IBL 확장 설계: 시간과 목적 개념

> IndieBiz OS — Phase 26 설계 계획서
> 2026년 3월

---

## 1. 배경과 동기

현재 IBL은 강력하지만 근본적인 한계를 가지고 있다. 에이전트는 명령을 받아 실행하고 끝낸다. 검색이 부족해도, 정보가 불완전해도 그냥 넘어간다. 완료의 기준이 없기 때문이다.

사용자가 기대하는 것은 다르다. '이 물건 살 때 제일 좋은 방법 찾아줘'라고 하면, 에이전트가 두루 살펴보고, 비교하고, 대안까지 검토한 다음 보고해주기를 원한다. 지금은 그게 안 된다.

원인은 에이전트가 두 가지를 모르기 때문이다.

- 이게 왜 필요한가 (목적)
- 언제까지 해야 하는가 (시간)

> 💡 목적과 시간은 에이전트를 도구에서 동료로 바꾸는 핵심 개념이다.

---

## 2. 핵심 개념

### 2.1 목적(Goal) 개념

목적은 에이전트에게 '왜'를 알려준다. 지금 IBL은 '어떻게'의 언어다. 목적이 들어오면 에이전트가 단순히 명령을 실행하는 게 아니라 목적 달성 여부를 스스로 판단한다.

목적이 있으면:

- 검색 결과가 불충분하면 스스로 더 찾는다
- 완료 기준을 에이전트가 판단한다
- 목적에 맞게 리소스(도구)를 배치한다
- 목적이 달성되면 보고하고 종료한다

### 2.2 시간(Time) 개념

시간은 에이전트에게 '언제까지'를 알려준다. 이것은 단순한 타이머가 아니다. 시간이 있으면 에이전트가 위임받은 존재가 된다.

시간 통제는 두 축으로 구성된다.

**종료 통제** — 언제 멈추는가:

- 시한 안에서 자율적으로 움직인다
- 시한에 따라 탐색의 깊이를 조절한다 (1시간 vs 1일)
- 시한이 되면 중간 결과라도 보고한다
- 미래 시점에 실행되도록 예약할 수 있다

**빈도 통제** — 얼마나 자주 하는가:

- 반복 실행 주기를 선언한다 (매일, 매주, 매시간 등)
- 빈도와 종료 조건이 결합되어 장기 모니터링을 구성한다
- 빈도에 따라 각 실행의 깊이가 자동 조절된다

> 💡 시간 개념은 요청-응답 구조를 위임 구조로 바꾼다. 매니저가 직원에게 '내일까지 알아봐'라고 하는 것, 또는 '매주 월요일마다 보고해'라고 하는 것과 같다.

### 2.3 조건(Condition) 개념

조건은 에이전트에게 '어떤 상황에서'를 알려준다. 목적과 시간이 합쳐진 상태에서 조건문이 들어오면 IBL은 명령 언어에서 정책 언어가 된다. 에이전트가 상황을 읽고 스스로 어떤 목적을 추구할지 결정한다.

## 3. 문법 설계

### 3.1 기존 IBL 문법

```
[node:action]{params}
[sense:apt_trade]{region: "청주"} >> [self:file]
```

기존 문법은 일회성 명령이다. 실행하고 끝난다.

### 3.2 목적 블록 (Goal Block)

목적은 IBL 실행의 상위 레이어로 선언된다.

```
[goal: "청주 부동산 투자 적기 판단"]{
  success_condition: "가격하락 + 금리하락 동시 감지",
  resources: ["real-estate", "investment", "local-info"],
  max_rounds: 100,
  max_cost: 5000,
  report_to: "사용자"
}
```

또는 기존 액션에 목적을 인라인으로 추가:

```
[sense:apt_trade]{region: "청주", goal: "투자 적기 판단", until: "조건달성"}
```

> 💡 모든 Goal Block에는 반드시 `max_rounds` 또는 `max_cost` 중 하나 이상이 포함되어야 한다. 무한루프 방지를 위한 필수 안전장치다.

### 3.3 시간 표현

시간 표현은 종료 통제와 빈도 통제 두 축으로 나뉜다.

**종료 통제 (언제 멈추는가):**

| 표현 | 의미 | 예시 |
|------|------|------|
| `deadline` | 최종 기한 | `deadline: "2026-12-31"` |
| `until` | 조건 달성까지 | `until: "매수결정"` |
| `within` | 기한 내 완료 | `within: "2h"` |
| `by` | 특정 시점까지 보고 | `by: "오늘 저녁"` |

**빈도 통제 (얼마나 자주 하는가):**

| 표현 | 의미 | 예시 |
|------|------|------|
| `every` | 반복 실행 주기 | `every: "매주 월요일 08:00"` |
| `schedule` | 일회성 예약 실행 | `schedule: "2026-04-01 09:00"` |

> 💡 `every`는 빈도를, `schedule`은 일회성 예약을 담당한다. 종료 통제와 빈도 통제를 조합하면 "매주 확인하되 6개월 안에 결론"과 같은 위임이 가능하다.

### 3.4 종료 우선순위

`until`, `deadline`, `max_rounds`, `max_cost`가 동시에 존재할 때의 종료 우선순위:

1. **`until` 충족** — 목적 달성. 즉시 종료하고 보고한다. `every` 다음 주기를 기다리지 않는다.
2. **`deadline` 도달** — 시한 만료. 현재까지 결과를 보고하고 `expired`로 전환한다.
3. **`max_rounds` / `max_cost` 도달** — 안전 한도. 현재까지 결과를 보고하고 `limit_reached`로 전환한다.

어떤 종료 조건이든 먼저 도달한 쪽이 적용된다.

### 3.5 안전 한도

모든 루프 실행에는 반드시 안전 한도가 포함된다.

| 표현 | 의미 | 예시 |
|------|------|------|
| `max_rounds` | 최대 반복 횟수 | `max_rounds: 100` |
| `max_cost` | 최대 비용 한도 (USD) | `max_cost: 3.0` |

**비용 산출 방식:**

- `max_cost`의 단위는 USD. 사용하는 모델의 토큰 단가를 기준으로 산출한다.
- 비용에는 판단 루프 자체의 LLM 호출 비용(success_condition 평가, 조건문 평가)도 포함된다.
- 외부 API 호출 비용은 각 패키지의 메타데이터에 선언된 호출당 예상 비용을 합산한다.

**사용자 확인 플로우:**

Goal 생성 시 에이전트는 예상 비용을 산출하여 사용자에게 확인을 받고 시작한다.

```
"이 Goal은 매일 실행 × 최대 200라운드로, 예상 비용은 약 $3.2입니다. 진행하시겠습니까?"
```

사용자가 승인하면 실행을 시작하고, 거부하면 Goal을 `cancelled`로 전환한다.

### 3.6 조건문 (if/else)

외부 상태나 sense 결과에 따라 다른 Goal을 활성화한다.

```
[if: sense:kospi < 2400]{
  [goal: "방어적 포트폴리오 재편"]{deadline: "즉시", max_rounds: 10}
} [else]{
  [goal: "성장주 모니터링 유지"]{every: "매일 09:00", max_rounds: 30}
}
```

파일 존재 여부, API 응답 값, 날씨, 주가 등 sense 노드가 읽을 수 있는 모든 것이 조건이 될 수 있다.

```
[if: sense:weather == "비"]{
  [goal: "실내 일정 최적화"]{by: "오늘 아침", max_rounds: 5}
} [else if: sense:weather == "맑음"]{
  [goal: "야외 활동 추천"]{by: "오늘 아침", max_rounds: 5}
} [else]{
  [goal: "일반 일정 유지"]{max_rounds: 3}
}
```

> 💡 조건문은 Goal과 결합할 때 가장 강력하다. 단순한 분기가 아니라 상황 인식 → 목적 설정의 흐름이 된다.

### 3.7 케이스문 (case)

여러 경우를 명확하게 분기할 때 if/else보다 가독성이 좋다.

**문자열 매칭:**

```
[case: sense:market_status]{
  "상승장": [goal: "공격적 매수 기회 탐색"]{deadline: "이번 주", max_rounds: 20},
  "하락장": [goal: "손절 라인 점검"]{deadline: "즉시", max_rounds: 10},
  "횡보": [goal: "배당주 모니터링"]{every: "매주 월요일", max_rounds: 12},
  default: [goal: "시장 관망"]{max_rounds: 5}
}
```

**범위 표현식 매칭:**

case문은 숫자 범위 비교를 지원한다. 파서가 처리하는 범위 표현식:

| 표현 | 의미 | 예시 |
|------|------|------|
| `> N` | N 초과 | `"> 20%"` |
| `>= N` | N 이상 | `">= 100"` |
| `< N` | N 미만 | `"< 10%"` |
| `<= N` | N 이하 | `"<= 50"` |
| `== N` | N과 같음 | `"== 0"` |
| `N~M` | N 이상 M 이하 | `"10~20%"` |

`%`는 sense 결과가 백분율임을 나타내는 표기이며, 파서는 숫자 부분만 추출하여 비교한다.

case문의 분기는 위에서 아래로 평가되며, 먼저 일치하는 분기가 실행된다. 어떤 분기에도 일치하지 않으면 `default`가 실행된다.

구매 의사결정 예시:

```
[case: sense:price_diff]{
  "> 20%": [goal: "즉시 구매 추천"]{by: "오늘", max_rounds: 5},
  "10~20%": [goal: "추가 비교 후 결정"]{within: "3h", max_rounds: 15},
  "< 10%": [goal: "대기 및 재모니터링"]{every: "매일", max_rounds: 30}
}
```

**동시 Goal과 case문:**

같은 맥락의 Goal이 충돌할 수 있는 경우, case문으로 분기하면 한 시점에 하나의 Goal만 활성화되므로 충돌이 원천 방지된다. 완전히 독립적인 Goal(예: 부동산 모니터링과 에어컨 구매)은 리소스가 겹치지 않으므로 병렬 실행을 허용한다.

### 3.8 통합 예시

구매 최적화 위임:

```
[goal: "에어컨 최적 구매"]{
  by: "오늘 저녁",
  resources: ["shopping-assistant", "web"],
  success_condition: "가격/성능/배송 비교 완료",
  max_rounds: 20,
  max_cost: 1000,
  report_to: "사용자"
}
```

> 💡 에이전트는 이 선언을 받고 시한까지 자율적으로 검색하고, 비교하고, 저녁에 보고한다.

부동산 모니터링 (장기):

```
[goal: "청주 투자 적기 감지"]{
  every: "매일 08:00",
  deadline: "2026-09-30",
  until: "매수 결정",
  resources: ["real-estate", "investment"],
  success_condition: "금리 하락 + 가격 10% 하락 동시",
  max_rounds: 200,
  max_cost: 50000
}
```

부동산 투자 전략 (조건 통합):

```
[goal: "청주 투자 적기 판단"]{
  every: "매일 08:00",
  deadline: "2026-09-30",
  until: "매수 결정",
  max_rounds: 200,
  max_cost: 50000,
  strategy: [case: sense:interest_rate]{
    "하락": [sense:apt_trade]{region: "청주", depth: "deep"},
    "상승": [goal: "관망"]{max_rounds: 1},
    default: [sense:apt_trade]{region: "청주", depth: "shallow"}
  }
}
```

> 💡 에이전트는 이 선언 하나로 금리 상황을 읽고, 그에 맞는 깊이로 탐색하고, 조건이 충족될 때까지 매일 반복한다.

## 4. 안전장치 설계

Goal 기반 루프 실행에는 세 겹의 안전장치가 적용된다.

### 4.1 내부 제한 (자동 브레이크)

모든 루프에 `max_rounds`와 `max_cost`가 필수 포함된다. 에이전트 스스로의 브레이크 역할을 한다.

- `max_rounds`: 판단 루프의 최대 반복 횟수. 도달 시 현재 결과 보고 후 종료.
- `max_cost`: 누적 비용 한도 (USD). 판단 루프 자체의 LLM 비용 + 액션 실행 비용 모두 포함. 도달 시 현재 결과 보고 후 종료.
- 종료 우선순위: `until` 충족 > `deadline` 도달 > `max_rounds`/`max_cost` 도달. 먼저 도달한 쪽이 적용된다.

### 4.2 목적 검증 (자체 검증)

Goal 생성 시 에이전트가 프롬프트 레벨에서 `success_condition`의 판단 가능성을 자체 검증한다.

검증 기준:

- 이 `success_condition`이 에이전트가 판단할 수 있는 조건인가?
- 정량적 조건(숫자 비교, 존재 여부)인가, 정성적 조건(좋은 결과, 충분한 조사)인가?
- 정성적 조건이라면 정량적 기준으로 변환할 수 있는가?

판단이 어려운 경우 사용자에게 명확화를 요청한다.

### 4.3 외부 통제 (프로세스 관리)

사용자가 진행 중이거나 예정된 프로세스를 언제든 조회하고 중단할 수 있다.

```
# 진행 중인 Goal 조회
[self:list_goals]{status: "active"}

# 예정된 Goal 조회
[self:list_goals]{status: "pending"}

# 특정 Goal 중단
[self:kill_goal]{goal_id: "goal_20260309_001"}

# 특정 Goal 상태 조회
[self:goal_status]{goal_id: "goal_20260309_001"}
```

> 💡 세 겹 안전장치: (1) 내부 제한 — 에이전트 스스로 멈춤, (2) 목적 검증 — 시작 전 판단 가능성 확인, (3) 외부 통제 — 사용자가 언제든 개입 가능.

## 5. 시스템 구조 변화

### 5.1 Goal 생명주기

목적은 생성되고, 살아있고, 달성되거나 소멸한다. 이 상태를 DB에서 관리한다.

| 상태 | 설명 |
|------|------|
| `pending` | 생성됨, 아직 시작 전 (예약 실행 대기) |
| `active` | 현재 에이전트가 추구 중 |
| `achieved` | `success_condition` 충족, 완료 보고됨 |
| `expired` | `deadline` 도달, 미달성으로 보고됨 |
| `limit_reached` | `max_rounds` 또는 `max_cost` 도달, 중간 결과 보고됨 |
| `cancelled` | 사용자가 취소 (`kill_goal`) |

### 5.2 기존 컴포넌트 활용

새로 만들 것이 많지 않다. 기존 컴포넌트들이 이미 조각으로 존재한다.

| 필요 기능 | 기존 컴포넌트 | 추가 작업 |
|-----------|--------------|-----------|
| 예약 실행 | scheduler | goal 상태와 연동 |
| 반복 판단 루프 | agent_runner | goal 달성 여부 판단 + 안전 한도 체크 추가 |
| IBL 자동 생성 | IBL RAG | goal 문맥 주입 |
| 상태 저장 | business.db / SQLite | goals 테이블 추가 |
| 보고 | self:notify_user | 그대로 활용 |
| 프로세스 관리 | — | list_goals, kill_goal, goal_status 액션 추가 |

### 5.3 판단 루프

에이전트가 목적을 추구하는 핵심 흐름:

1. Goal 생성 → 예상 비용 산출 → 사용자 확인 → Goal 활성화
2. 종료 조건 체크 (우선순위): `until` 충족 → `deadline` 도달 → `max_rounds`/`max_cost` 도달
3. 현재 상태 평가: `success_condition` 충족 여부 확인 (이 평가 비용도 `max_cost`에 합산)
4. 미충족이면: 조건문 평가 → IBL 생성 → 실행 → 결과 평가 → 라운드 결과 DB 저장 → 2로 복귀
5. 충족이면: 보고 → Goal 상태를 `achieved`로 변경
6. `deadline` 초과 시: 현재까지 결과 보고 → `expired`
7. 안전 한도 초과 시: 현재까지 결과 보고 → `limit_reached`

**시스템 재시작 시:** 실행 중이던 라운드는 포기한다. Goal 상태는 DB에서 `active`로 유지되며, 다음 `every` 주기에 자동 재개된다. `every`가 없는 일회성 Goal은 재시작 후 마지막 체크포인트부터 재개한다.

> 💡 이 루프는 agent_runner의 멀티 라운드 실행과 구조가 같다. Goal이 라운드 종료 조건을 제공하고, `max_rounds`/`max_cost`가 비상 브레이크를 제공하는 것이다.

## 7. 기대 효과

### 7.1 에이전트 질적 변화

| 항목 | 현재 | Goal/Time/Condition 도입 후 |
|------|------|---------------------------|
| 역할 | 도구 (명령 실행) | 동료 (목적 추구) |
| 완료 기준 | 없음 (실행하면 끝) | 목적 달성 여부로 판단 |
| 탐색 깊이 | 한 번 검색하고 끝 | 충분할 때까지 반복 |
| 시간 인식 | 없음 | 시한과 빈도에 따라 행동 조절 |
| 보고 | 즉시 응답 | 적시 보고 (by 저녁, every 매주 등) |
| 상황 인식 | 없음 | 조건에 따라 전략 변경 |
| 안전 | 없음 | 세 겹 안전장치 (내부 제한 + 목적 검증 + 외부 통제) |

### 7.2 사용자 경험 변화

지금은 사용자가 에이전트를 계속 감독해야 한다. 부족하면 다시 시키고, 더 찾아보라고 하고.

Goal/Time/Condition이 들어오면 한 번 위임하고 기다리면 된다. 에이전트가 알아서 하고, 때가 되면 보고한다. 너무 오래 걸리면 프로세스 목록에서 확인하고 중단할 수 있다.

> 💡 이것이 진짜 위임이다. 결과가 아니라 목적을 맡기는 것. 그리고 언제든 상황을 확인하고 개입할 수 있는 것.

## 8. 구현 고려사항

| 항목 | 내용 |
|------|------|
| 조건 평가 시점 | Goal 활성화 시, 그리고 각 판단 루프 반복 시 |
| sense 결과 캐싱 | 동일 조건을 매 루프마다 재조회하지 않도록 TTL 설정 필요 |
| 중첩 깊이 제한 | 조건문 안에 조건문이 과도하게 중첩되면 디버깅이 어려워짐 (AI 생성이므로 가독성보다 실행 안정성 중심) |
| 파서 확장 | ibl_parser.py에 goal/if/else/case/every 토큰 + 범위 표현식 파서 추가 필요 |
| 오류 처리 | 조건 평가 실패 시 default 분기 또는 에러 보고 |
| 비용 추적 | 모델별 토큰 단가 기반 비용 산출. 판단 루프 LLM 비용 + 액션 실행 비용 합산. 외부 API 비용은 패키지 메타데이터의 호출당 예상 비용 참조 |
| 비용 확인 | Goal 생성 시 예상 비용을 산출하여 사용자에게 확인 후 시작. "약 $X 예상, 진행하시겠습니까?" |
| 중간 체크포인트 | 장기 Goal의 경우 각 라운드 결과를 DB에 저장하여 중단 시에도 부분 결과 보존 |
| 시스템 재시작 복구 | 실행 중이던 라운드는 포기하고, 다음 `every` 주기를 기다려 재개. Goal 상태는 DB에서 `active` 유지 |
| 동시 Goal 실행 | 같은 맥락의 Goal은 case문으로 분기하여 충돌 방지. 독립적인 Goal은 병렬 실행 허용 |
| 범위 표현식 | case문에서 `> N`, `< N`, `>= N`, `<= N`, `== N`, `N~M` 지원. 정규식으로 패턴 분리 후 숫자 비교 |

## 9. 설계 원칙

- **목적 달성 판단 로직이 핵심이다.** success_condition을 AI가 어떻게 해석하는지가 품질을 결정한다. 프롬프트에서 판단 가능성을 자체 검증하도록 한다.
- **모든 루프에 안전 한도가 있다.** `max_rounds`와 `max_cost`는 선택이 아니라 필수다.
- **사용자가 언제든 개입할 수 있다.** 진행 중인 프로세스 조회와 중단이 가능해야 한다.
- **기존 IBL과의 호환성을 유지한다.** Goal Block과 조건문은 선택적 레이어다.
- **IBL은 AI가 작성하는 언어다.** 문법 복잡도보다 표현력이 우선이다.

---

## 10. 결론

목적, 시간, 조건문이 합쳐진 IBL은 세 가지 질문에 모두 답하는 언어가 된다.

- 무엇을 위해 (Goal)
- 언제까지 / 얼마마다 (Time)
- 어떤 상황에서 (Condition)

이 세 가지가 갖춰지면 에이전트는 명령을 받는 존재에서 정책을 이해하는 존재가 된다. 사용자가 상황마다 지시하지 않아도, 에이전트가 상황을 읽고 스스로 목적을 설정하고 추구한다. 세 겹의 안전장치가 이 자율성을 통제 가능하게 만든다.

기술적 난이도는 높지 않다. 기존 컴포넌트들이 조각으로 존재하며, Goal 레이어, 조건문 파서, 프로세스 관리 액션 추가가 핵심이다.

## Phase 26b: 전략 전환 규칙 + 라운드 메모리

> Phase 26의 Goal/Time/Condition과 별개로, 에이전트의 실행 패턴 자체를 개선하는 두 가지 메커니즘.

### 배경

Phase 26 이전의 실행 기록을 분석한 결과, 에이전트가 67라운드 동안 같은 유형의 시도를 미세 변형하며 반복하는 패턴이 확인되었다. Goal/Time/Condition은 "왜, 언제까지, 어떤 조건에서"를 해결하지만, "어떻게 실패에 대응하는가"는 별도 레이어가 필요하다.

### 11. 전략 전환 규칙 (Strategy Escalation)

에이전트의 시스템 프롬프트에 `<strategy_rules>`로 주입되는 행동 규칙:

1. **매 시도 후 기록**: `[self:log_attempt]`로 접근 범주, 설명, 결과, 배운 점을 구조화하여 저장
2. **3회 연속 실패 시 전환**: 동일 `approach_category`가 3회 연속 실패하면 그 범주를 포기하고 근본적으로 다른 접근으로 전환
3. **모든 범주 소진 시 보고**: 시도 가능한 모든 접근 범주가 실패하면 사용자에게 상황을 보고하고 판단을 요청
4. **이력 확인 후 시도**: 새 시도 전에 `[self:get_attempts]`로 이전 이력을 확인하여 같은 실수를 반복하지 않음
5. **백업 후 수정**: 파일을 수정하기 전에 현재 상태를 백업하여 복원 가능성 확보

### 12. 라운드 메모리 (Attempt History)

에이전트가 이전 시도를 구조화하여 기억하는 메커니즘.

**DB 구조 (attempt_log 테이블):**

| 필드 | 설명 |
|------|------|
| `task_id` | 같은 작업의 시도를 묶는 키 |
| `agent_id` | 실행 에이전트 |
| `round_num` | 자동 증가하는 시도 번호 |
| `approach_category` | 접근 범주 (예: "cv2_import", "pillow_fallback") |
| `description` | 구체적으로 무엇을 시도했는지 |
| `result` | "success" 또는 "failure" |
| `lesson` | 이 시도에서 배운 점 |

**컨텍스트 주입:**

시스템 프롬프트의 `<attempt_history>` 섹션으로 현재 태스크의 최근 시도 이력이 동적 주입된다. 포기된 범주는 `<exhausted_categories>`로 명시되어 에이전트가 해당 접근을 반복하지 않도록 한다.

**IBL 액션:**

```
[self:log_attempt]{task_id: "task_001", approach_category: "cv2_import", description: "cv2 직접 임포트 시도", result: "failure", lesson: "cv2 미설치"}
[self:get_attempts]{task_id: "task_001"}
```

**자동 경고 시스템:**

`[self:log_attempt]` 호출 시 연속 실패 횟수를 자동 계산하여:
- 3회 연속 실패: `escalation_required: true` + 범주 전환 지시
- 모든 범주 소진: `all_exhausted: true` + 사용자 보고 지시

### 13. 구현 파일

| 파일 | 변경 내용 |
|------|-----------|
| `conversation_db.py` | `attempt_log` 테이블 + CRUD 4개 메서드 |
| `ibl_nodes.yaml` | self 노드에 `log_attempt`, `get_attempts` 액션 추가 |
